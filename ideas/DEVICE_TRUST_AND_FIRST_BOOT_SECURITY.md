# Device Trust + First-Boot Security

## Why this exists

Harden platform access beyond username/password by introducing device trust approval, bootstrap admin safety, and token-gated Web UI onboarding.

## Problem

- Default admin bootstrap is fragile and may not reliably match runtime auth expectations.
- First login security posture is weak if default credentials are unchanged.
- Any new device can attempt Web UI access without explicit trust approval flow.

## Proposed Direction

### 1) Bootstrap admin creation at install/start

- Ensure installer/startup bootstrap always creates the initial admin account if missing.
- Treat bootstrap as idempotent (safe on restarts).
- Keep explicit warning state when default credentials are still active.

### 2) Post-login security warning for default admin credentials

- After admin login, show a clear warning popup if default credentials are detected.
- Include exact destination where password/username can be changed.
- Keep warning until credentials are changed.

### 3) Device approval gate (OpenClaw-like trust model)

- New devices must be approved by admin before full access.
- Access requires:
  - valid user token
  - approved device identity
- Device identity should be generated by system and displayed at login prompt.

### 4) First approval path after install

- First trust approval must happen locally via TUI/admin command (sudo path).
- After first trusted admin session exists, future approvals can happen in Admin UI.

### 5) Admin UI approvals table

- Provide pending device table with share-safe metadata only:
  - device/connection id
  - first-seen timestamp
  - client version
  - rough platform metadata (OS/browser class)
  - source IP (if policy allows)
- Actions: approve, deny, revoke existing trust.
- All actions should be audit logged.

### 6) Channels handling

- Keep channel-specific trust behavior as a follow-up design decision.
- Some channels may require differentiated policy due to external identity constraints.

## Constraints / Security Notes

- No silent auto-approval for unknown devices.
- Approval flow must be auditable and reversible.
- Metadata shown in approval UI should exclude sensitive/private data by default.

## Suggested Rollout

1. Reliable bootstrap admin + default-credential warning.
2. Device identity issuance + server-side trust store.
3. First local TUI approval requirement.
4. Admin UI approval table + token/device combined checks.
5. Channel-specific policy extension.
