# v2.6.0 Plugin Ecosystem — Design Spec

**Date:** 2026-05-12
**Milestone:** v2.6.0 — Plugin Ecosystem (Java-First)
**Status:** Approved

---

## 1. Goal

Deliver a robust, secure, extensible Java plugin system for SYNAPSE. Plugins extend the platform
without touching core code — admins install a JAR, the system loads it, agents use it. The
foundation must be strong enough that every future plugin type builds on it without architectural
changes.

**In scope for v2.6.0:** Plugin API module, Plugin Loader, Plugin Lifecycle Manager, Dependency
Resolver, Sandboxing, CLI tooling (full operator set), Dashboard marketplace UI, and the first
two plugin categories: **Channels** and **Model Providers**.

**Deferred to v2.7.0:** Skills, MCP Servers, Plugin Registry Service, External Plugin Runtime
(Python/Node.js), Official Plugin Library expansion.

## 3. Plugin API Module (`synapse-plugin-api`)

The single JAR all plugin developers depend on. Plugins cannot import anything outside it —
enforced at runtime via JPMS module layers.

### 3.1 Interfaces

| Interface | Purpose |
|---|---|
| `SynapsePlugin` | Base for all plugin types — `getId()`, `getManifest()`, `onLoad()`, `onUnload()` |
| `Channel` | Bidirectional messaging — already exists, minor cleanup only |
| `ModelProvider` | LLM backend — `complete()`, `stream()`, `getCapabilities()` |
| `PluginContext` | Injected at load time — scoped access to logger, config, event bus, agent router |
| `PluginConfig` | Typed wrapper around manifest `config_schema` values |
| `PluginEventBus` | Plugin-to-core event publishing |
| `PluginLogger` | Scoped logger — all output tagged with plugin id, routed to system log |

### 3.2 What plugins cannot access

Spring beans, JPA repositories, Redis clients, internal service classes — anything outside
`synapse-plugin-api`. The JPMS `ModuleLayer` for each plugin only has `requires synapse.plugin.api`.
Core modules are not exported to plugin layers. Violation = `ClassNotFoundException` at load time.

### 3.3 Distribution

- Published to **GitHub Packages** as `dev.synapse:synapse-plugin-api:<version>`
- Semantic versioned independently of SYNAPSE core
- Plugin manifests declare `requires.api_version: ">=1.0.0"`
- Migration path: move to own Nexus/Sonatype instance in a future release

### 3.4 Plugin Template

- Repo: `/synapse-plugin-template` — Gradle-based, includes example Channel and ModelProvider
- `synapse plugin scaffold` CLI command creates a new GitHub repo from this template
- Template declares `synapse-plugin-api` as the only compile dependency

## 4. Plugin Storage & Lifecycle

### 4.1 Storage Layout

```
$SYNAPSE_HOME/
├── plugins/
│   ├── system/          ← persisted across restarts
│   │   ├── telegram-channel-1.2.0.jar
│   │   └── anthropic-provider-1.0.0.jar
│   └── staging/         ← runtime-installed this session
│       └── discord-channel-0.9.0.jar
```

### 4.2 Staging → System Migration

| Event | Behavior |
|---|---|
| Runtime install | JAR placed in `staging/`, soft-reloaded immediately |
| Graceful shutdown | All `staging/` JARs migrated to `system/` |
| Hard crash / kill | `staging/` JARs remain on next start; admin prompted to confirm promote or discard |
| Clean startup | Only `system/` JARs loaded automatically |

### 4.3 Startup Load Sequence

```
1. Scan system/ for JARs
2. Parse + validate each manifest
3. Run dependency resolver (hard deps first, then soft)
4. Conflict check across all JARs
5. For each JAR: create isolated URLClassLoader + JPMS ModuleLayer
6. Inject PluginContext, call onLoad()
7. Register in ChannelRegistry or ModelProviderRegistry
8. Mark ACTIVE in DB
```

### 4.4 Soft-Reload (Runtime Install)

```
store install command
  → download JAR → place in staging/
  → validate manifest
  → dependency resolver (auto-install hard deps from store if missing)
  → conflict check against loaded plugins
     → conflict? → reject JAR, clear from staging, send diagnostic to admin
     → clean?    → load into new ClassLoader + ModuleLayer
  → call onLoad(), register in registry
  → mark ACTIVE in DB
```

### 4.5 Config Update

Plugin config can be updated at runtime without uninstalling. On save:
1. Validate new config against `config_schema`
2. If new **required** field added by an update: block, prompt admin to fill it in first
3. Call `onUnload()` on running instance
4. Re-inject `PluginContext` with new config, call `onLoad()`
5. No full application restart needed

## 5. Conflict Detection & Version-Aware Resolution

### 5.1 Conflict Rules

| Scenario | Behavior |
|---|---|
| Same plugin id, same version | Block — already installed |
| Same plugin id, **newer** version | Prompt admin to update |
| Same plugin id, **older** version | Block — newer already active |
| Two plugins claim same `channel_id` or `provider_id` slot | Hard block |
| Incompatible `api_version` range | Block at install |

### 5.2 Update Flow

`synapse plugin update <id>`:
1. Validate new JAR manifest
2. If `config_schema` changed: check for new required fields → prompt admin if needed
3. Call `onUnload()` on old instance
4. Swap ClassLoader to new JAR
5. Inject updated `PluginContext`, call `onLoad()`
6. Move new JAR to correct storage tier

### 5.3 Error Message Format

```
Plugin conflict: cannot load [discord-channel v1.1.0]
Reason: plugin id "discord-channel" already active (v1.0.0)
Currently loaded: discord-channel v1.0.0 (system, loaded at startup)
Resolution: run `synapse plugin update discord-channel` to upgrade.
```

```
Plugin update available: discord-channel v1.0.0 → v1.1.0
Run `synapse plugin update discord-channel` to apply.
```

## 6. Dependency Resolution

### 6.1 Manifest Declaration

```yaml
requires:
  synapse_version: ">=2.6.0"
  api_version: ">=1.0.0"
  java: ">=25"
  plugins:
    - id: oauth-helper
      version: ">=1.0.0"
      type: hard

soft_requires:
  - id: metrics-exporter
    version: ">=1.0.0"
    feature: "Enables per-plugin Prometheus metrics export"
```

### 6.2 Hard Dependency Flow

```
Install plugin X
  → read requires.plugins[]
  → for each hard dep:
      → already installed + compatible?  → continue
      → installed but wrong version?     → trigger update flow
      → not installed?                   → auto-fetch from store → install recursively
      → not in any store?                → BLOCK, list all missing deps clearly
```

### 6.3 Soft Dependency Behavior

- Missing soft dep → plugin loads fully, optional feature silently absent
- Soft dep later installed → plugin notified via `PluginEventBus` → feature activates, no reload needed
- Dashboard shows "optional feature unlocked" notification

### 6.4 Circular Dependency Guard

Resolver builds a directed dependency graph before any installs begin. Cycle detected → block
entire install, report full cycle chain:

```
Dependency cycle detected:
  plugin-a → plugin-b → plugin-c → plugin-a
Cannot install. Break the cycle by removing one dependency.
```

## 7. Plugin Sandboxing & Security

Three enforcement layers, applied in order.

### Layer 1 — Install-Time Bytecode Scan

Before any JAR is loaded, ASM walks every class file. Rejects JARs that reference:
- `sun.*`, `com.sun.*`
- `java.lang.reflect.AccessibleObject`
- Spring internals (`org.springframework.beans.*`, etc.)
- Hibernate/JPA internals
- Any class not in `synapse-plugin-api` or the JDK standard library

Output format:
```
✗ Bytecode violation: com.example.MyPlugin line 42 — forbidden ref: sun.misc.Unsafe
✗ Bytecode violation: com.example.Util line 17 — forbidden ref: org.springframework.beans.factory.BeanFactory
Result: REJECTED (2 violations). Fix these references before installing.
```

### Layer 2 — JPMS Module Isolation (Runtime)

Each plugin JAR loaded into its own `ModuleLayer`. Layer only has `requires synapse.plugin.api`.
Core modules not exported to plugin layers. Any class outside `synapse-plugin-api` throws
`ClassNotFoundException` — no bypass path.

### Layer 3 — Resource Limits via PluginContext

Plugin threads created only via `PluginContext.executor()` — a bounded virtual thread pool per
plugin:

| Limit | Default (Community) | Default (Official) | Configurable |
|---|---|---|---|
| Concurrent virtual threads | 10 | 25 | Per plugin in manifest |
| Lifecycle hook timeout (onLoad, onUnload) | 10s | 30s | Per plugin in manifest |
| Message handler timeout | 30s | 60s | Per plugin in manifest |
| Log volume | 1000 lines/min | 5000 lines/min | Global setting |

Exceeding limit: throttled + admin dashboard warning. Lifecycle hook timeout exceeded: plugin
marked `ERROR`, disabled automatically.

### 7.1 Trust Tier Influence

| Tier | Bytecode Scan | Resource Defaults | Store Badge |
|---|---|---|---|
| Official (`/synapse-plugins`) | Mandatory | Relaxed | Blue — Official |
| Community (`/synapse-plugins-community`) | Mandatory + stricter | Strict | Yellow — Community |

Community plugins pass CI/CD gates in `/synapse-plugins-community` repo (automated tests,
manifest validation, bytecode scan) before appearing in the store.

## 8. CLI Tooling

All commands in `/packages/cli`, built on the existing Bubble Tea TUI framework.

### 8.1 Full Command Set

```
synapse plugin scaffold      # TUI wizard → creates GitHub repo from /synapse-plugin-template
synapse plugin validate      # validate manifest + run bytecode scan, report violations
synapse plugin package       # compile + package distributable JAR with manifest baked in
synapse plugin install <id>  # install from store (official or community)
synapse plugin update <id>   # update installed plugin to latest compatible version
synapse plugin list          # table: id, name, type, version, status, trust tier
synapse plugin enable <id>   # enable disabled plugin (soft-reload)
synapse plugin disable <id>  # disable without uninstalling
synapse plugin uninstall <id># graceful onUnload() + remove JAR from staging/system
synapse plugin info <id>     # full detail: manifest, deps, soft deps, resource usage, trust tier
synapse plugin logs <id>     # Bubble Tea live log view scoped to plugin id
synapse plugin publish       # STUB — planned, deferred. Prints guidance message.
```

### 8.2 `scaffold` TUI Flow

```
? Plugin name: _______________
? Plugin type: [Channel] / [Model Provider]
? Author: _______________
? License: [MIT] / Apache-2.0 / other
→ Creates GitHub repo from /synapse-plugin-template
→ Prints repo URL + next steps (build, validate, package)
```

### 8.3 `publish` Stub

`synapse plugin publish` in v2.6.0 prints:
```
Plugin publishing is coming in a future release.
For now, submit your plugin to:
  Community:  https://github.com/FTMahringer/synapse-plugins-community
```

## 9. Dashboard Marketplace & Plugin Management UI

### 9.1 Store Browse Page

- Grid/list toggle of available plugins
- Filter by: type (Channel, Model Provider), trust tier, compatibility with current Synapse version
- Search by name or tag
- Each card: name, author, description, trust tier badge, version, install count

### 9.2 Plugin Detail Page

- Full description, version history, changelog per version
- Dependency tree (hard + soft deps, clickable)
- Trust tier badge with explanation tooltip
- Install button → inline install flow with progress indicator
- If already installed: current vs latest version, Update button if newer exists

### 9.3 Install Flow

```
Click Install
  → dependency check shown: "Will also install: oauth-helper v1.2.0"
  → admin confirms
  → progress: downloading → validating → scanning → loading
  → success: plugin moves to Installed tab
  → conflict: inline error with resolution hint + update prompt if applicable
```

### 9.4 Installed Plugins Management

- Table: name, type, version, status (ACTIVE / DISABLED / ERROR), trust tier, storage (system / staging)
- Per-row actions: Enable, Disable, Update, Uninstall, View Logs, View Info
- ERROR status: inline diagnostic message, link to logs

### 9.5 Config Update UI

- Config form generated from manifest `config_schema`
- Secret fields masked, never shown after first save
- On save: validates, soft-reloads plugin
- If update adds new required field: form blocks save until field filled
