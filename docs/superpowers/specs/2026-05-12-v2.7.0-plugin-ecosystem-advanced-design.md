# v2.7.0 Plugin Ecosystem Advanced Features — Design Spec

**Date:** 2026-05-12
**Milestone:** v2.7.0 — Plugin Ecosystem Advanced Features
**Status:** Approved
**Depends on:** v2.6.0 Plugin Ecosystem (Java-First)

---

## 1. Goal

Expand the plugin ecosystem built in v2.6.0 with a robust Plugin Registry Service, four new plugin
types (MCP Server, Skill, Skill Bundle, Bundle), expanded official plugin library, full governance
layer, and admin security controls. v2.7.0 completes the plugin platform story for self-hosters
and companies alike.

**In scope for v2.7.0:**
- Plugin Registry Service (embedded + standalone, multi-source)
- Custom registry sources (GitLab, Forgejo, Nexus/Maven, private)
- Store Policy admin controls (community filtering, CLI command restrictions)
- MCP Server plugin type (declarative YAML)
- Skills + Skill Bundle plugin type (skills.sh wrapper)
- Bundle (meta) plugin type (cross-type install packages)
- Governance: plugin signing, reporting/takedowns, deprecation + compatibility matrix
- Official plugin library expansion (mix of all 4 types)

**Deferred to v2.8.0:** External Plugin Runtime Foundation (process manager, gRPC IPC).
**Deferred to v2.9.0:** External language runtimes (Python, Node.js).

## 3. Plugin Registry Service

### 3.1 Deployment Modes

Configured via `SYNAPSE_REGISTRY_MODE` env var:

| Mode | Description | Use case |
|---|---|---|
| `embedded` (default) | Registry runs inside SYNAPSE backend process | Self-hosters, single instance |
| `standalone` | Own container, SYNAPSE backend is a client via `SYNAPSE_REGISTRY_URL` | Companies, multi-instance |

Same codebase — one Spring Boot module, toggled by config. Standalone mode exposes a documented
REST API.

### 3.2 Registry Sources

Each source has a **type** and **trust tier**:

| Source | Type | Trust |
|---|---|---|
| `github.com/FTMahringer/synapse-plugins` | `github` | Official |
| `github.com/FTMahringer/synapse-plugins-community` | `github` | Community |
| `gitlab.mycompany.com/plugins` | `gitlab` | Private |
| `forgejo.mycompany.com/plugins` | `forgejo` | Private |
| `nexus.mycompany.com/repository/synapse` | `maven` | Private |

Admin registers custom sources via dashboard (Settings → Plugin Sources) or CLI:
```
synapse registry add-source \
  --name "Company Internal" \
  --type gitlab \
  --url https://gitlab.mycompany.com/plugins \
  --token ${GITLAB_TOKEN} \
  --visibility store    # or: internal
```

**Visibility options:**
- `internal` — admin CLI/dashboard only, never shown in store UI
- `store` — appears in marketplace as a labeled install source

Private sources scope to that instance only — never federated to community registry.

### 3.3 Core Capabilities

**Metadata API:**
- Search by name, tag, type, trust tier, compatibility with current Synapse version
- Version history per plugin
- Compatibility matrix: which plugin versions work with which Synapse versions
- Deprecation flags + migration hints

**Artifact Proxy + Cache:**
- JARs fetched from source once, cached locally
- Cache TTL configurable; manual invalidation available
- On install: serve from cache; fallback to source on miss
- Signature verification before serving (see Governance section)

**Source sync:**
- Periodic background sync from all registered sources
- Configurable sync interval per source
- Manual trigger: `synapse registry sync` or dashboard button

### 3.4 Store Policy Controls

Configured in dashboard (Settings → Plugin Store Policy):

```yaml
plugin_policy:
  scaffold:
    allowed_roles: [admin, developer]
  publish:
    allowed_roles: [admin]
  install:
    allowed_roles: [admin, user]
  community_plugins:
    enabled: true           # false = community tab hidden, installs blocked
    whitelist: []           # if non-empty: only matching plugins shown
    blacklist: []           # always hidden/blocked regardless of whitelist
```

Whitelist/blacklist operate on: plugin id, author, tag, trust badge — any combination.
When a blocked user runs a restricted CLI command:
```
Permission denied: plugin scaffold is not allowed for your role.
Contact your SYNAPSE administrator to request access.
```

## 4. New Plugin Types

### 4.1 MCP Server Plugin Type

Declarative YAML only — no Java code, no ClassLoader, no bytecode scan. Plugin registers an
external MCP server endpoint with SYNAPSE core. Core handles the MCP protocol.

**manifest.yml:**
```yaml
id: filesystem-mcp
type: mcp
version: 1.0.0
author: synapse-team

mcp:
  transport: stdio
  command: "npx"
  args: ["@modelcontextprotocol/server-filesystem", "/data"]
  # For sse/http transport:
  # url: "https://mcp.example.com"
  env:
    API_KEY: "${config.api_key}"

config_schema:
  api_key:
    type: string
    required: false
    secret: true
```

Plugin loader detects `type: mcp`, parses manifest, registers with `MCPServerRegistry`. No
security sandboxing needed — the MCP server is an external process.

### 4.2 Skills Plugin Type

Wrapper format around skills.sh skills. SYNAPSE fetches each skill from skills.sh on install,
stores locally, registers with agent runtime.

**Single skill:**
```yaml
id: web-search-skill
type: skill
version: 1.0.0

skills:
  - source: skills.sh
    id: web-search
    version: ">=2.0.0"
```

**Skill bundle:**
```yaml
id: research-skills-bundle
type: skill-bundle
version: 1.0.0

skills:
  - source: skills.sh
    id: web-search
    version: ">=2.0.0"
  - source: skills.sh
    id: summarize
    version: ">=1.0.0"
  - source: skills.sh
    id: cite-sources
    version: ">=1.5.0"
```

Install: fetches all skills, stores locally, registers as a unit. Uninstall: removes all skills
in bundle atomically. Admin sees bundle as a single entry in dashboard. No ClassLoader involved.

### 4.3 Bundle (Meta) Plugin Type

Cross-type meta-manifest. Groups plugins of any type into a single installable unit.

```yaml
id: research-powerpack
type: bundle
version: 1.0.0
author: synapse-team
description: "Everything you need for AI-powered research workflows."

includes:
  - id: telegram-channel
    version: ">=1.2.0"
    source: official
  - id: anthropic-provider
    version: ">=1.0.0"
    source: official
  - id: brave-search-mcp
    version: ">=1.0.0"
    source: official
  - id: research-skills-bundle
    version: ">=1.0.0"
    source: official

requires:
  synapse_version: ">=2.7.0"
```

**Bundle install behavior:**
- Dependency + conflict resolution runs for all components before any install begins
- Trust tier = lowest trust tier of any component (one Community component = bundle is Community)
- If one component blocked by store policy: entire bundle install blocked; clear message names the component
- Dashboard shows bundle as single card with expandable component list
- One-click install, one-click uninstall (removes all components installed by this bundle)

## 5. Governance Layer

Three independent sub-systems, each in its own dev version.

### 5.1 Plugin Signing & Verification

Authors sign JARs and manifests with a private keypair before publishing. Registry verifies on
ingest and again on every install.

**CLI flow:**
```
synapse plugin sign telegram-channel-1.2.0.jar
  → uses author keypair from ~/.synapse/keys/
  → signature embedded in JAR manifest
  → registry ingests → verifies → stores fingerprint
```

**Trust badges:**

| State | Badge | Install behavior |
|---|---|---|
| Signed + verified (Official) | Blue — Verified Official | Default allow |
| Signed + verified (Community) | Green — Verified Community | Default allow |
| Unsigned (Community) | Yellow — Unverified | Warning + admin confirm |
| Signature mismatch | Red — Tampered | Hard block, no override |

### 5.2 Reporting & Takedown Workflow

Users flag plugins from the store detail page or CLI (`synapse plugin report <id>`).

**Report categories:** malicious, broken, license violation, spam.

**Flow:**
```
User reports plugin
  → stored in registry DB
  → admin notified (dashboard + optional email)
  → admin reviews: dismiss / warn author / takedown
  → takedown: hidden from store, existing installs get dashboard warning
  → auto-hide threshold: configurable (e.g. 5 reports in 24h → pending review)
```

For self-hosted registries: admin is sole reviewer.
For `/synapse-plugins-community`: GitHub issue auto-created for maintainer review.

### 5.3 Deprecation & Compatibility Matrix

**Manifest deprecation field:**
```yaml
deprecated:
  since: "2.1.0"
  reason: "Use discord-channel-v2 instead."
  migration: "https://docs.example.com/migrate"
```

**Compatibility matrix** (registry-maintained):
```
telegram-channel:
  1.0.0 → synapse >=2.6.0 <2.8.0
  1.1.0 → synapse >=2.7.0
  1.2.0 → synapse >=2.7.0  ← current
```

Store + CLI warn on deprecated version or incompatible Synapse version. Update prompt shown
automatically. Admin can suppress warnings instance-wide for frozen environments via
`plugin_policy.suppress_deprecation_warnings: true`.
